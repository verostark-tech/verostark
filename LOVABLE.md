# Verostark — Lovable Frontend Spec

## Overview

Verostark detects deviations between what a STIM royalty statement paid and what
a Nordic music publisher's registered catalogue entitles them to receive. The app
explains every deviation in plain English and tells the administrator exactly what
to do next.

**Users:** Copyright administrators at Nordic music publishers.
**Language:** English only for V0.1.
**Auth:** Clerk magic-link, organisation-scoped. Every API call requires a Clerk
          JWT passed as `Authorization: Bearer <token>`.

---

## API client — use this, never hand-write fetch calls

The file `src/lib/client.ts` is generated by Encore and is the only way to call
the backend. Do not write `fetch("/api/...")` manually anywhere.

Initialise once at the app root:

```typescript
import Client, { Environment } from "@/lib/client"
import { useAuth } from "@clerk/clerk-react"

const { getToken } = useAuth()

export const backend = new Client(
  Environment("staging"),         // swap to Environment("prod") when going live
  {
    auth: async () => {
      const token = await getToken()
      return { authorization: `Bearer ${token}` }
    }
  }
)
```

All screens import `backend` from this shared module. No component should
instantiate its own client.

Severity badge colours:

- `HIGH` → orange

## Screen 2 — Upload

**Route:** `/upload`

**Purpose:** Upload a STIM royalty statement CSV for deviation analysis.

### Layout

Single upload card, centred.

**Title:** Upload STIM statement

**Explainer text:**
> Upload your STIM royalty statement CSV. Once uploaded, run the deviation check
> from the Statements screen.
> Accepted format: .csv

**Fields:**
- File picker (accept `.csv` only)
- Period (text input, placeholder "e.g. 2025-Q1") — required

**Flow (exact sequence — do not deviate):**

```
Step 1: POST /files/upload   (multipart/form-data, field name "file")
        → { key, filename, size, mime_type }

Step 2: POST /api/statements  body { file_key: key, filename, period }
        → Statement object

On success: green banner "Statement registered — run the deviation check from
            the Statements screen." with a link to /statements.
On error:   show the `message` field from the API JSON response in a red banner.
```

**Important:** `file_key` from Step 1 must be included in Step 2's request body.
The API will reject the call without it.

## Screen 3 — Statements

**Route:** `/statements`

**Purpose:** List all uploaded statements. Run the deviation check. Navigate to results.

### Layout

Page header: "Statements"

Table: all statements from `GET /api/statements`
Columns: Filename | Period | Status | Uploaded | Actions

**Actions column:**

- **Status `pending`:** "Run detection" button.

  On click:
  1. Disable the button, replace label with "Running…" and a spinner.
  2. Call `POST /api/statements/:id/run`.
  3. This call can take **15–30 seconds** (AI explanations are generated server-side).
     Keep the spinner visible for the full duration. Do not timeout before 60 seconds.
  4. On success: toast "Detection complete — {flag_count} deviations,
     {unmatched_count} unmatched lines." Refresh the row status to `completed`.
  5. On error: toast with the `message` field from the API response. Re-enable button.

- **Status `processing`:** disabled "Running…" button with spinner.

- **Status `completed`:** "View deviations" link → `/statements/:id/deviations`

### 4a — Deviations list

**Data:**
- `/statements/:id/deviations` → `GET /api/deviations?statement_id=:id`
- `/deviations` → `GET /api/deviations`

**Filters bar:**
- Severity dropdown: All | LOW | MEDIUM | HIGH | CRITICAL
- Status dropdown: All | open | resolved
- Re-fetch with updated query params on change.

**Table columns:**
Work title | ISWC | Right type | Period | Expected (SEK) | Received (SEK) | Deviation | Severity | Status

Format currency as `{value} SEK` (2 decimal places).
Deviation: show signed percentage. Red if negative (underpayment), amber if positive (overpayment).
Each row is clickable → `/deviations/:id`

**Empty state:** "No deviations found."

## Error handling

- Non-2xx response → show `response.message` in a red toast.
- Network unreachable → "Unable to reach the server. Check your connection."
- Never show stack traces or technical error codes.
- If `RunDetection` is still in-flight after 60 seconds, show "This is taking longer
  than expected. The server is still working — do not close this tab."

---

## Loading states

- Skeleton loader (grey shimmer) for tables while data loads.
- Spinner on buttons during async actions.
- Disable all interactive elements during in-flight requests.
- `RunDetection` spinner must persist for the full duration (up to 60 seconds).
